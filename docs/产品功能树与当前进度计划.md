# 医宝：产品功能树与当前进度计划

## 1. 项目概述

医宝是一个以“AI 陪诊”为核心的医疗就医助手，目标不是单点问诊，而是把用户从症状表达、分诊判断、医院推荐、挂号约诊、到院执行、院中服务、诊后管理串成一条完整链路。

当前设计重点优先放在：

- 就医助手核心主链路
- 其中优先级最高的是：`预问诊 + 分诊推荐`
- 第二阶段继续补齐：`医院推荐 + 挂号约诊`
- 后续逐步完善：`到院协助 + 院中服务 + 医疗档案 + 诊后提醒`

## 3. 当前设计进度

### 3.1 已完成

#### 产品结构梳理

- 已完成整体功能架构拆解
- 已形成产品功能树
- 已梳理用户主链路与模块关系

#### 就医助手主链路：第一阶段

已完成 `预问诊 + 分诊推荐` 的优化方案定义，设计方向为：

- 温和陪伴型
- 延续原稿暖橙主色
- 保持当前字体气质和医疗场景的轻柔感
- 从“流程机”体验优化为“私人健康顾问”体验

已细化页面包括：

1. 问诊起始页
2. 温和追问页
3. 初步判断页
4. 科室建议页
5. 下一步引导页

#### 就医助手主链路：第二阶段预览

已继续补齐以下页面方向：

1. 医院推荐页
2. 挂号偏好收集页
3. 号源推荐页

#### 就医助手主链路：第三阶段预览

已开始补齐 `到院协助` 模块，当前方向为：

- 全流程接驳助手
- 同时支持老人本人操作与家属代操作
- 默认以老人自己操作为主，界面表达更直接、按钮更大、信息分组更清楚

已细化页面包括：

1. 到院路线选择页
2. 打车确认页
3. 到点提醒页

当前这部分设计重点为：

- 不只解决“怎么去”，还解决“会不会迟到、到了后怎么办”
- 路线选择更直接，对比打车 / 公交 / 自驾三种方式
- 打车场景尽量减少输入，强调“一键确认”
- 增加停车提示、迟到风险和到点提醒等老年友好信息

#### 就医助手主链路：第四阶段预览

已继续补齐 `院中服务 + 诊后承接` 模块，当前已细化页面包括：

1. 缴费提醒页
2. 化验结果提醒页
3. 回科室继续就诊页
4. 取药提示页
5. 病历归档页
6. 护理建议页
7. 用药提醒页

当前这部分设计重点为：

- 院中服务优先处理“眼前最需要执行的一步”，减少老人和家属自己判断
- 化验结果优先转化成“下一步做什么”，而不是直接堆叠报告内容
- 就诊结束后自动把诊断、检查和处方聚合为一张可回看的病历卡
- 诊后护理和用药提醒采用“少量、分天、可执行”的表达方式，降低执行负担

#### 就医助手主链路：第五阶段预览

已继续补齐 `异常态 + 确认态`，当前已细化页面包括：

1. 挂号确认页
2. 支付成功页
3. 支付失败页
4. 检查结果未出页
5. 复诊提醒页

当前这部分设计重点为：

- 关键确认节点优先展示最必要信息，避免误操作和重复确认
- 成功态要快速把用户带回主流程，不停留在“结果页”
- 失败态要给出替代路径，不让用户在中途掉线
- 等待态和复诊态都要转化成可执行的提醒任务，减少用户自己记忆成本

#### 就医助手主链路：第六阶段预览

已继续补齐 `高价值边角状态`，当前已细化页面包括：

1. 检查结果部分已出页
2. 挂号取消 / 改约页
3. 到院超时提醒页
4. 家属代操作确认页
5. 院内带路页

当前这部分设计重点为：

- 让常见但容易被忽略的分支状态也有明确承接，不把用户丢回系统外自己判断
- 到院超时、部分已出等等待型状态都要转化成下一步可执行动作
- 家属代操作要明确权限边界、任务归属和提醒接收对象
- 院内带路强调楼层、方向和目的地，而不是复杂地图阅读

## 8. 结论
